
 Remains to be Fixed (Gaps & Improvements)
While the core is solid, there are several critical areas that need attention:
A. Security Vulnerabilities (High Priority)
Permissive Firestore Rules: The current firestore.rules allow create and update on the qrCodes collection for anyone (allow create, update: if true;). This means a malicious user could overwrite the ownerId of any QR code and "steal" its branding.
PII Exposure: User profile rules are too broad. Any authenticated user can potentially read another user's profile document if they have the ID, which could leak sensitive information if not carefully filtered.
Backend Rate Limiting: The /api/qr/decode-image endpoint lacks rate limiting, making it vulnerable to Denial of Service (DoS) attacks User Account Management: Missing features like "Delete Account," "Export Data," or a comprehensive "Scan History" management (currently only shows recent scans). Code Quality & Maintenance
Silent Failures: Many try-catch blocks in firestore-service.ts have empty catch blocks. This hides errors and makes it extremely difficult to diagnose issues when a database operation fails.
Project Duplication: The QRS directory appears to be a near-duplicate of scan-guard. This creates confusion and should be merged or removed to ensure a single source of truth.
