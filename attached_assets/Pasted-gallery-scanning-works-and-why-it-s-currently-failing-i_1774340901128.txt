1. How the Gallery Scan Works (The Current "Broken" Flow)
When a user clicks the "Gallery" button, the app follows these steps:
Image Selection: The app uses a "Picker" to open your phone's photo library. Once you select a photo, the phone gives the app a Local File URI (a path like file:///var/mobile/...).
The Decoding Attempt: To read the QR code from that photo, the app needs to send that image data to a "decoder."
The "Fetch" Command: Currently, the app is using a standard web command called fetch() to try and "grab" the image from your phone's storage to process it.
The Failure: Standard fetch() is designed for the internet (URLs starting with http), not for local phone files (starting with file://).
The Error: Because fetch() doesn't understand the local file path, it fails immediately. The app sees this failure and shows you a generic "Network Error: Could not reach server" message, even though the problem isn't the internet—it's just the app failing to read its own local file.
2. Why the Camera "Crashes" the App
If a phone's camera is physically broken or blocked, the app crashes because of a "Hardware Conflict":
Permission vs. Availability: The app currently checks if it has permission to use the camera. If you say "Yes," it immediately tries to turn the camera hardware on.
The Crash: If the hardware is broken, the phone's operating system sends back an error that the app isn't prepared to handle. Instead of showing a "Camera not found" message, the app "panics" and closes (crashes).
The Solution Concept: The app needs to separate the Scanner Screen from the Camera Hardware. It should check if the camera is working before trying to show the video feed. If it's not working, it should just show the "Upload from Gallery" button on a plain background.
3. The "Network Error" Explained
The reason you see "Could not reach server" for every gallery upload is due to Path Misinterpretation:
Real Devices vs. Simulators: This often works on a developer's computer (simulator) but fails on a real phone. Real phones have strict security that prevents standard web commands from touching local files.
The "False" Network Error: When the internal command fails to read the image, it triggers a "Network Request Failed" exception. The app's error handler is programmed to say "Could not reach server" whenever any request fails, leading to the confusing message you are seeing.
Summary of the Conceptual Fix:
For the Gallery: Instead of using fetch() to read the file, the app should use a dedicated File System tool (like expo-file-system) to convert the image into a "Base64 String" (a long text version of the image). Sending text is 100% reliable and won't trigger network errors.
For the Camera: The app should "wrap" the camera in a safety check. If the camera fails to start, the app stays open and simply tells the user: "Camera unavailable. Please use the Gallery option below."
For the Error Message: The app needs a smarter "Error Detector." If the problem is a local file issue, it should say "Could not read image," and if it's a real internet issue, it should say "Server unreachable."
