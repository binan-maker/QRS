 A core feature is broken right now
The gallery QR scan in features/scanner/hooks/useScanner.ts line 299 uses globalThis.fetch — the wrong network function for Expo. It will fail with "Network request failed" on most real devices. This is unacceptable in a production app.

3. The rate limiter will reset every time the server restarts
In server/routes.ts, the rateLimitMap is a plain JavaScript Map stored in memory. Every time the Express server restarts, all rate limit data is wiped. A production app uses Redis or a database for this.   No error monitoring exists
There is no Sentry, no Crashlytics, no logging service anywhere in the code. When a real user's app crashes, you will never know it happened. In production, you need to know about every crash within seconds.

5. Firebase free tier will break the app
Firebase Firestore free tier allows 50,000 reads per day. With 1,000 active users each opening the app 5 times and loading 3–4 screens, that is already 750,000+ reads per day — 15× over the limit. The entire app goes blank. There is no Firebase billing setup, no usage monitoring, no alerts. 


7. The server has zero authentication on the image decode endpoint
In server/routes.ts, the /api/qr/decode-image endpoint checks a token header but does not reject requests without one. Any person on the internet can spam this endpoint with images and consume your server resources, even without an account.

No automated tests
Zero test files exist in the project. Every change you make can silently break something you do not know about until a real user reports it.   The app cannot run without Expo Go
The package.json shows "main": "expo-router/entry" and the build scripts show expo start --localhost. This means users cannot download a standalone app from Play Store or App Store. They must first download Expo Go, then scan a QR code to open your app. Every normal user will stop at this step. A production app has its own standalone .apk or .ipa file.
